# 문서 통합 설계

**목표:** 흩어져 있던 리뷰, 흐름, 개선 문서를 현재 시스템과 요청 흐름을 설명하는 두 개의 기준 문서로 통합한다.

## 범위

- `docs/architecture-overview.md`를 프로젝트 구조, 도메인 모델, API 규칙, 인프라, 테스트, 현재 리스크를 설명하는 단일 기준 문서로 만든다.
- `docs/project-flow.md`를 요청 생명주기와 엔드포인트별 비즈니스 흐름을 설명하는 단일 기준 문서로 만든다.
- 더 이상 필요 없는 `docs/`와 `.docs/`의 문서를 제거한다.

## 접근 방식

### 1. 독자의 목적에 따라 분리

- `architecture-overview.md`는 "이 프로젝트가 무엇이고, 어떻게 구성되며, 어떤 규칙으로 동작하는가?"에 답한다.
- `project-flow.md`는 "요청이 시스템을 어떻게 통과하며, 각 API는 어떤 흐름으로 동작하는가?"에 답한다.

현재 문서들은 정적 구조, 리뷰 이력, 엔드포인트 흐름이 한데 섞여 있어 유지보수가 어렵기 때문에 한 파일로 합치지 않는다.

### 2. 이력이 아니라 현재 상태를 기준으로 정리

- 문서는 현재 코드와 맞는 내용만 남긴다.
- 과거 리뷰 결과는 시간순 기록으로 남기지 않고, 필요한 경우 짧은 "남은 리스크 / 후속 작업" 섹션으로 흡수한다.
- `.docs/*`와 기존 `docs/*`의 대체본은 유지하지 않는다.

### 3. 운영에 필요한 정보는 유지

- 응답 형식, 인증, 페이지네이션, 도메인 관계, 동시성 보호 방식은 유지한다.
- Docker/CI 설명은 현재 구조를 이해하는 데 필요한 수준으로 요약한다.
- 이미 해결된 과거 이슈보다 아직 살아 있는 리스크만 남긴다.

## 문서 구성

### `docs/architecture-overview.md`

- 프로젝트 목적
- 기술 스택과 앱 경계
- 실행 규칙
- 도메인 모델 요약
- API 구성 요약
- 동시성과 데이터 무결성 메모
- 인프라와 테스트 전략
- 남은 리스크와 다음 우선순위

### `docs/project-flow.md`

- 전역 요청 생명주기
- 인증 흐름
- 상품 관련 흐름
- 위시리스트 흐름
- 입찰 흐름
- 주문 생성 및 포인트 이동 흐름
- 주문 상태 변경 흐름
- 입찰 취소 흐름
- 마이페이지 및 시세 히스토리 흐름

## 검증 기준

- 기존 문서의 핵심 내용이 새 기준 문서 두 개에 반영되었는지 확인한다.
- 삭제 대상 문서가 실제로 완전히 대체되었는지 확인한다.
- 최종 `docs/` 구조만 봐도 어디서 읽기 시작해야 하는지 분명한지 확인한다.
